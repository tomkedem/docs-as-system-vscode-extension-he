# מדריך יישום מלא  

Docs-as-System-he

## מבוא

מסמך זה הוא מדריך יישום מפורט למי שפותח פרויקט חדש בעזרת הרחבת Docs-as-System ל-VS Code.  
המטרה שלו היא להוביל אותך שלב אחרי שלב, משלב יצירת הפרויקט ועד עבודה שוטפת עם הסוכן.  
המסמך כתוב בצורה ישירה וברורה כדי שתוכל להתניע את השיטה בלי שאלות פתוחות או לחץ מיותר.

המדריך בנוי כמו מסלול ברור.  
כל שלב מסתמך על השלב שקדם לו.  
בסוף המסלול הפרויקט שלך יהיה מתועד, מסודר ויציב.

## התחלת עבודה בפרויקט חדש

הגעת לכאן אחרי שהקמת פרויקט חדש דרך הרחבת Docs-as-System ל-VS Code.
בשלב זה הפרויקט כבר מוכן וכולל את כל מבנה התבניות, התיקיות והקבצים הדרושים.

## נוצר פרויקט חדש שמוכן להתחלה מידית

הפרויקט כולל:

• שלד בסיסי של קוד  
• תיקיית docs מלאה בתבניות  
• קובצי תצורה של הסוכן  
• סקריפטים שמנהלים פעולות Git בצורה מבוקרת  
• קובץ README ראשוני  

כדי להתחיל נכון:

נכנסים לתיקייה של הפרויקט החדש  
פותחים את תיקיית docs  
עובדים לפי סדר השלבים שמופיע בהמשך

# חלק שני

זהו השלב הקריטי שבו מניחים את היסודות של כל פרויקט.

מומלץ לכתוב מסמך זה בעזרת מודל שיחה חיצוני.  
הוא מסייע לנסח מחשבות בצורה מסודרת וברורה בלי לבזבז זמן.

## שלב רביעי  
### Architecture Decision Records (ADR)

בשלב זה מתעדים החלטות ארכיטקטוניות חשובות שמעצבות את המערכת.  
החלטה שלא תועדה – לא קיימת עבור הסוכן ואינה חלק רשמי מהארכיטקטורה.

עובדים לפי מבנה קבוע וברור:

---

### 📁 היכן נמצאים ה־ADRs?

כל ADR נשמר כקובץ נפרד בתיקייה:

```
docs/architecture/decisions/
```

---

### 🧩 שם הקובץ – חובה לשמור על הפורמט

שמירת סדר ויכולת חיפוש תלויה במבנה שם הקובץ:

```
ADR-YYYY-NN_<נושא>.md
```

לדוגמה:

```
ADR-2025-09_Monolith_vs_Microservices.md
```

- YYYY — שנת כתיבת ההחלטה  
- NN — מספר רץ באותה השנה  

המספור הרץ חיוני כדי לשמור על סדר, עקביות ואפשרות מעקב.

---

### 📚 קובץ אינדקס לכל ההחלטות

כל ADR חייב להופיע בקובץ אחד מרכזי:

```
docs/architecture/decisions/ADR_INDEX.md
```

האינדקס מאפשר:

- לראות את מפת ההחלטות של המערכת  
- להימנע מכפילויות  
- לספק לסוכן הקשר מלא  
- לאתר החלטות במהירות  

האינדקס כתוב לפי התבנית:

```
MethodologyTemplates/architecture/ADR_INDEX_TEMPLATE.md
```

---

### 📝 איך כותבים ADR?

הכתיבה מבוצעת לפי התבנית הרשמית:

```
MethodologyTemplates/architecture/ADR_TEMPLATE.md
```

בכל ADR מתועדים:

• מה הוחלט  
• למה הוחלט  
• אילו חלופות נשקלו  
• מה ההשלכות  
• מדדים שיאפשרו לבדוק אם ההחלטה הייתה מוצלחת  
• קשר למסמכים אחרים בפרויקט  

ההשקעה כאן חוסכת טעויות עתידיות ומאפשרת לסוכן לעבוד בהקשר אמיתי.

---

### 🤖 האם כדאי להשתמש במודל שיחה חיצוני?

כן.  
הוא עוזר לחדד את הנימוקים, לחשוב על חלופות, ולכתוב בצורה עקבית וברורה.  
עם זאת – חשוב לעקוב אחרי התבנית ולא לשבור את המבנה שלה.

---

### 🎯 למה השלב הזה חשוב?

- הוא יוצר "זיכרון ארכיטקטוני" אמיתי  
- הוא מונע החלטות סותרות  
- הוא מאפשר לסוכן להבין את הכוונות הארכיטקטוניות  
- הוא מבטיח שכל שינוי במערכת תואם את המודל התכנוני  

זהו אחד השלבים הקריטיים ביותר ב־Docs-as-System.

## AGENT_OPERATIONAL_POLICY  
המסמך נכתב על ידי אדם שאחראי על תהליך העבודה בפרויקט. בדרך כלל זה יהיה ארכיטקט, מנהל פרויקט או טכנולוג ראשי.  
המטרה של המסמך היא לקבוע עבור הסוכן מערכת חוקים ברורה שמשמשת כחוזה עבודה מחייב.

### מי כותב  
אדם בכיר שמכיר את תהליך העבודה, את מדיניות הארגון ואת גבולות האחריות של הסוכן.  
הסוכן אינו כותב את המסמך ואינו משנה אותו.

### איך כותבים  
מנסחים את הכללים בצורה ישירה וברורה.  
רצוי להשתמש במודל שיחה חיצוני כדי ליצור ניסוחים עקביים ולוודא שהמסמך לא משאיר מקום לפרשנות.

### מה כולל  
• פעולות שמותר לסוכן לבצע ללא אישור  
• פעולות שמותר לבצע רק לאחר אישור אנושי  
• תהליך עצירה במקרה של חריגה או כשל  
• ציפיות לגבי איך הסוכן צריך לטפל בשגיאות  
• כללים לתיעוד בלוגים ולדיווח בזמן אמת  
• כללים להגבלת גישה לקבצים, תיקיות או משאבים רגישים  
• הנחיות לתקופתיות של עדכון המסמך

### מתי כותבים  
לאחר סיום שלבי התכנון (planning ו architecture), ולפני שהסוכן מתחיל לבצע פעולות בפועל בפרויקט.

### איך הסוכן משתמש במסמך  
הסוכן קורא את המסמך בכל התחלה של תהליך ובכל הפעלה של פעולה רגישה.  
אם פעולה חורגת מהחוקים שמוגדרים כאן, הסוכן עוצר את הביצוע ומבקש אישור מפורש.

### מי כותב  
אדם שמכיר היטב את אופן פעולת הסוכן ואת תהליך העבודה בפרויקט.

### איך כותבים  
מנסחים הנחיות קצרות, בהירות ופרקטיות.

### מתי כותבים  
לאחר הגדרת המדיניות של הסוכן.

### איך הסוכן משתמש במסמך  
הסוכן אינו קורא את המסמך הזה.  
הוא מיועד לבני אדם בלבד ומבטיח שההוראות שניתנות לסוכן יהיו עקביות, בטוחות וברורות.

# תיקיית automation

שכבת האוטומציה אחראית להפעלת תהליכים אחידים ובטוחים לאורך חיי הפרויקט.  
היא מוודאת שכל פעולה טכנית שמבוצעת על ידי הסוכן או על ידי האדם תהיה מתועדת, עקבית ונשלטת.

בתיקיית automation תמצא מסמכים שמגדירים את הזרימה של התהליכים, וגם סקריפטים שמבצעים פעולות בפועל.

## WORKFLOWS_GUIDE

זהו מדריך שמסביר איך לבנות תהליכים עקביים.  
הוא כולל:

• שלבי חיים של תהליך  
• טריגרים להפעלה  
• פעולות שהסוכן מפעיל  
• נקודות עצירה  
• דרישות לדיווח בלוגים  

המסמך מסייע להבין איך אוטומציה צריכה להתנהג בפרויקט שמנוהל על פי השיטה.

---

## VALIDATION_PIPELINE

זהו תיאור של צנרת אימות.

• מה התנאים להמשך התהליך

ה Pipeline אינו ספציפי לפלטפורמה מסוימת.  
הוא מתאר לוגיקה בלבד כדי שכל פרויקט יוכל ליישם אותה בכל מערכת CI.

## תיקיית git

תיקייה זו כוללת סקריפטים לביצוע פעולות Git בצורה מבוקרת ובטוחה.  
הסוכן לעולם לא מפעיל פקודות ישירות במסוף.  
הוא משתמש רק בקבצים שבתיקייה זו.

הסקריפטים מאפשרים:

• יצירת ענפים  
• הכנת שינויים  
• יצירת קומיט  
• דחיפת ענפים  
• פתיחת בקשות מיזוג  
• מיזוג לאחר אישור אנושי  

כל סקריפט מתועד ומחזיר תוצאה אחידה ללוגים.

# חלק שישי  
# מחזור עבודה מלא  
# איך עובדים בצורה רציפה ובטוחה

שיטת Docs as System עובדת במחזורים.  
כל מחזור כולל תכנון, ביצוע, תיעוד ובקרה.

להלן הסדר המקובל:

---

## פתיחת מחזור חדש

פותחים מחזור חדש בקובץ BY_CYCLE.  
מגדירים בו את:

• תאריכי המחזור  
• מטרות  
• משימות  
• נקודות בקרה

המסמך מעניק כיוון ברור לתקופה הקרובה.

---

## ביצוע עבודה עם הסוכן

עובדים על קוד ותיעוד.  
הסוכן מבצע פעולות בהתאם למדיניות והוא מעדכן את הלוגים אוטומטית.

האדם:

• נותן הוראות  
• בודק תוצאות  
• מאשר שינויים  
• עוצר ריצות במקרה הצורך

---

## תיעוד כל פעולה

במהלך המחזור כל פעולה חייבת להופיע ב IMPLEMENTATION_LOG.  
הלוג משמש כמקור אמת למשך חיי הפרויקט.

---

## הפקת דו"ח אימות

הסוכן מייצר VALIDATION_REPORT שמסכם את כל מה שבוצע.  
הדוח כולל הערכת איכות ופעולות שנותרו פתוחות.

האדם עובר על המסמך ומאשר אותו.

---

## יצירת תקציר Summary

זהו תקציר ניהולי קצר בסוף כל מחזור.  
הוא מסכם:

• מה הושלם  
• מה לא הושלם  
• מצב בדיקות  
• רמת סיכון  
• הערות מנהליות

---

## עדכון ה PLAN לקראת המחזור הבא

התוכנית מתעדכנת לפי מה שבוצע.  
לאחר העדכון מתחיל מחזור חדש.
