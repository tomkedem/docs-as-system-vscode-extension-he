## 🎯 מטרת הפרויקט

הסבר קצר וברור (בפסקה אחת או שתיים)  
על הסיבה לקיומו של הפרויקט ומה הוא נועד להשיג.

**דוגמה:**  
> המערכת נועדה לשפר את ניהול זרימת הנתונים הארגונית  
> באמצעות תיעוד חי ו-AI Agent שמבצע פעולות תחת בקרה אנושית.

## 🏗️ מבנה המאגר

תיאור כללי של תיקיות הפרויקט ותפקידיהן.

docs/
├── project/ ← זהות, תאימות ומדיניות הפרויקט
├── planning/ ← דרישות עסקיות, אפיון לוגי ותוכנית עבודה
├── architecture/ ← מפת ארכיטקטורה והחלטות הנדסיות (ADR)
├── agent/ ← מדיניות פעולה ותצורה לסוכן ולאדם
├── logs/ ← לוגים ומעקב לפי מחזורי עבודה (BY_CYCLE)
└── automation/ ← סקריפטים, דוחות ותהליכי אוטומציה
src/ ← קוד המקור של המערכת
tests/ ← בדיקות יחידה, אינטגרציה וולידציה

## 🧩 מחזור החיים של פרויקט Docs-as-System

כל פרויקט בשיטה זו מתנהל במבנה עקבי ושקוף של שכבות:

1. **Intent (planning)** — הגדרת הכוונה העסקית והיעדים.  
2. **Design (architecture)** — תכנון מבנה המערכת והחלטות מפתח.  
3. **Execution (agent + logs)** — ביצוע מתועד בזמן אמת.  
4. **Validation (automation)** — בדיקות ואימות אוטומטיים.  
5. **Governance (project)** — בקרה, תאימות וניהול גרסאות.

> כל שינוי — אנושי או אוטומטי — חייב להיות מתועד, נבדק ומאושר  
> לפני שהוא מתמזג לענף מוגן (Protected Branch).

## 🔄 ניהול גרסאות ובקרה

- ניהול גרסאות לפי **SemVer** (Semantic Versioning).  
- כל גרסה משויכת למזהה מחזור עבודה (`CycleId`).  
- הסוכן מבצע Commits מקומיים בלבד.  
- רק אדם מוסמך מאשר Merge לענפי הפצה.  
- כל שינוי מתועד ב־`IMPLEMENTATION_LOG.md`.
